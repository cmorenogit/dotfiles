# producto — lente para cuestionar

Aplica el conocimiento de producto para **cuestionar la premisa**, no para elegir entre las opciones que trae el dev. El deliverable es el insumo de producto; no decide scope (eso es de Nicole/Ignacio).

## Flujo

1. **Cargá el conocimiento:** `~/Code/_vault/_work/apprecio/_shared/producto.md` (canónico; engram `producto` como cache — si difieren, gana el vault).
2. **Clasificá el objeto:** propuesta/análisis de producto → checklist completo · PR/cambio técnico → kernel (6 preguntas) · respuesta liviana → kernel mental, reportá solo si algo dispara. Sin decisión de producto ni riesgo (typo, acuse) → no fuerces el canon.
3. **Cuestioná la premisa** (no elijas entre las opciones dadas): ¿cuál es el **outcome real**? ¿la solución está **sobredimensionada** para ese outcome? ¿cae sola del outcome, o se partió de la solución y se justificó hacia atrás? Separá el outcome (qué se quiere lograr) de la solución propuesta — el caso típico: el dev pide elegir entre A y B cuando el outcome real no necesitaba ninguna.
4. **Aplicá el kernel K1–K6** siempre; el checklist de 20 para propuestas a fondo.
5. **Fraseá como insumo** para el dueño del scope ("esto le falta a la propuesta para que Nicole/Ignacio decida X"), nunca como veredicto de alcance. Respetá T5: la validación de producto es de Ignacio, etapa previa.

## Salida

Hallazgos de producto: gap · evidencia (cita o ausencia) · para quién. Si el alcance está mal, la recomendación va como decisión binaria (mantener vs recortar), no como menú.
